Supply Chain Risk Management

Continuous assessment of third-party, fourth-party, and software supply chain risk posture across all vendor relationships.

→

→

→

→

▼
Feeds → Correlation Engine

📥 Inputs

  • Vendor/supplier inventory (name, service category, data access level)
  • SBOMs (Software Bill of Materials) for all third-party software
  • SOC 2 / ISO 27001 / penetration test reports from vendors
  • Vendor security questionnaire responses (SIG, CAIQ, custom)
  • External attack surface data on vendor domains and IPs
  • Threat intel feeds mentioning vendor breach indicators
  • Contract SLA and data processing agreement metadata
  • Fourth-party dependency maps (vendor's vendors)

📤 Outputs → Correlation Engine

  • Vendor risk scores — composite rating per vendor (critical/high/medium/low)
  • SBOM vulnerability map — CVEs inherited through third-party components
  • Concentration risk alerts — single-point-of-failure vendor dependencies
  • Breach exposure indicators — vendors with active breach signals
  • Contractual gap flags — missing DPA clauses, expired certifications
  • Fourth-party risk cascades — downstream dependency risk chains

⚙️ Process Steps

  • 1. Inventory & Classification — Maintain a living registry of all vendors. Classify each by data sensitivity (what data they touch), business criticality (impact if unavailable), and access level (network, API, physical).
  • 2. Risk Tiering — Assign each vendor to a tier (Tier 1: critical/high-data-access, Tier 2: moderate, Tier 3: low). Tier determines assessment frequency and depth.
  • 3. Initial & Periodic Assessment — Tier 1: full security assessment (questionnaire + evidence review + pen test results) annually. Tier 2: questionnaire + automated scanning semi-annually. Tier 3: automated external scanning quarterly.
  • 4. SBOM Ingestion & CVE Mapping — Parse SBOMs from vendor software. Cross-reference every component against NVD, OSV, and proprietary vuln databases. Flag critical/high CVEs with no vendor patch timeline.
  • 5. Continuous External Monitoring — Automated scanning of vendor external attack surfaces (SSL posture, open ports, DNS health, data leak detection, darkweb mentions). Feed anomalies into risk score.
  • 6. Incident Response Linkage — When a vendor appears in threat intel or breach databases, auto-escalate. Trigger assessment of exposure scope and contractual notification requirements.
  • 7. Remediation Tracking — Issue risk findings to vendor. Track remediation commitments with deadlines. Escalate overdue items. Accept risk with documented business justification where remediation is not feasible.

▼
Feeds → Correlation Engine

📥 Inputs

  • Cloud provider APIs (AWS Config, GCP Security Command Center, Azure Defender)
  • Infrastructure-as-Code repositories (Terraform, CloudFormation, Pulumi)
  • CSPM (Cloud Security Posture Management) scan results
  • IAM policies, roles, service accounts, and access keys across all providers
  • Network topology — VPCs, security groups, firewall rules, peering configs
  • Container/Kubernetes cluster configs (EKS, GKE, AKS)
  • On-premise asset inventory (CMDB, network scanners)
  • Cloud spend and resource utilization data (orphaned/zombie resources)

📤 Outputs → Correlation Engine

  • Asset risk inventory — every resource with risk score, owner, exposure level
  • Misconfiguration findings — CIS benchmark violations with severity and blast radius
  • IAM risk map — over-privileged accounts, stale credentials, lateral movement paths
  • Network exposure report — publicly exposed services, insecure ingress/egress
  • Drift detection alerts — IaC vs. actual state divergences
  • Multi-cloud risk comparison — normalized risk scores across providers

⚙️ Process Steps

  • 1. Multi-Cloud Asset Discovery — Enumerate all resources across AWS, GCP, Azure via APIs. Correlate with IaC repos to identify unmanaged (shadow) resources. Classify by business function, data sensitivity, and environment (prod/staging/dev).
  • 2. Configuration Baseline Assessment — Evaluate every resource against CIS Benchmarks, provider best practices, and organization-specific policies. Score each finding by exploitability, data exposure risk, and remediation complexity.
  • 3. IAM Deep Analysis — Map all identity-to-permission paths. Identify: over-privileged roles, unused permissions, cross-account trust chains, service account key rotation gaps, and MFA gaps. Build lateral movement graph.
  • 4. Network Exposure Mapping — Analyze all ingress/egress rules. Identify publicly reachable services, unrestricted CIDR blocks, missing WAF/DDoS protections, and VPN/peering misconfigurations. Overlay with known vulnerability data.
  • 5. Container & Kubernetes Posture — Scan cluster configs for RBAC misconfigurations, privileged containers, missing network policies, unpatched node images, and exposed dashboards/APIs.
  • 6. Drift Detection & Alerting — Continuously compare running infrastructure against IaC definitions. Flag manual changes, unauthorized modifications, and configuration regressions. Auto-create remediation tickets.
  • 7. Risk Aggregation & Owner Assignment — Roll up findings into per-resource and per-account risk scores. Map to asset owners. Generate prioritized remediation queues based on business criticality × vulnerability severity × exposure.

▼
Feeds → Correlation Engine

📥 Inputs

  • Network vulnerability scanners (Nessus, Qualys, Rapid7)
  • Application security testing — SAST, DAST, IAST results
  • Container image scanning (Trivy, Snyk Container, Prisma)
  • Open source dependency scanning (SCA) — Snyk, Dependabot, Grype
  • Cloud-native vulnerability findings (AWS Inspector, GCP SCC)
  • Penetration test and red team findings
  • Bug bounty submissions
  • NVD, EPSS, KEV catalog, vendor advisories

📤 Outputs → Correlation Engine

  • Unified vulnerability inventory — deduplicated, normalized across all scanners
  • Risk-ranked vulnerability queue — scored by CVSS × EPSS × asset criticality × exposure
  • Exploit availability flags — CVEs with known PoC/weaponized exploits
  • Remediation SLA status — on-track, at-risk, breached per severity tier
  • Attack path overlays — vulns that chain to create exploitable attack paths
  • Mean-time-to-remediate (MTTR) metrics — per team, severity, and asset class

⚙️ Process Steps

  • 1. Multi-Source Discovery — Run scheduled and triggered scans across network, application, container, and code layers. Normalize findings into a common vulnerability data model (CVE, affected asset, severity, evidence).
  • 2. Deduplication & Enrichment — Deduplicate findings across overlapping scanners. Enrich each CVE with: EPSS probability score, KEV catalog status, public exploit availability, vendor patch status, and asset business context.
  • 3. Risk-Based Prioritization — Score = f(CVSS base, EPSS probability, exploit maturity, asset criticality, network exposure, compensating controls). Rank into: Critical (fix in 24h), High (7d), Medium (30d), Low (90d).
  • 4. Automated Ticket Generation — Create remediation tickets in the engineering team's system (Jira, ServiceNow). Include: affected asset, CVE details, fix guidance, priority, SLA deadline, and asset owner assignment.
  • 5. Remediation Tracking — Monitor ticket progress against SLAs. Auto-escalate approaching and breached SLAs to engineering leads and then to CISO. Track exception/risk-acceptance requests with documented business justification.
  • 6. Verification Scanning — After reported fix, trigger targeted rescan to confirm remediation. If vuln persists, reopen ticket with failure evidence. Only close on verified fix.
  • 7. Metrics & Trending — Calculate MTTR, SLA compliance rate, vulnerability density per asset class, and month-over-month trending. Feed into executive dashboards and correlation engine.

Feeds → Correlation Engine

📥 Inputs

  • Commercial threat intel feeds (Recorded Future, Mandiant, CrowdStrike)
  • ISAC/ISAO industry-specific threat sharing
  • MITRE ATT&CK framework TTPs mapped to org's industry
  • SIEM/SOAR alert data and SOC investigation findings
  • EDR/XDR telemetry and behavioral detections
  • Email gateway threat data (phishing campaigns, BEC attempts)
  • Network traffic analytics (DNS, NetFlow, proxy logs)
  • User and entity behavior analytics (UEBA) anomalies

📤 Outputs → Correlation Engine

  • Active threat actor profiles — groups targeting the org's industry + their TTPs
  • Threat-to-asset mapping — which assets are targeted by which threat actors
  • IOC feeds — curated indicators of compromise for detection systems
  • Detection coverage matrix — ATT&CK techniques with/without detection rules
  • Active campaign alerts — ongoing campaigns affecting the org or peers
  • Threat trend analysis — evolving adversary behavior patterns

⚙️ Process Steps

  • 1. Intelligence Collection & Curation — Aggregate threat data from commercial feeds, open sources, ISACs, and internal telemetry. Filter for relevance to org's industry, geography, technology stack, and threat model.
  • 2. Threat Modeling — Maintain living threat models per critical business process. Map likely adversaries, their TTPs (using MITRE ATT&CK), target assets, and attack paths. Update quarterly and on major architecture changes.
  • 3. Detection Engineering — Write and tune detection rules mapped to ATT&CK techniques. Track detection coverage across the matrix. Prioritize rule development for techniques used by threat actors in step 2. Test rules against purple team exercises.
  • 4. Proactive Threat Hunting — Develop hunt hypotheses based on intel and gaps in detection coverage. Execute hunts using EDR, SIEM, and network data. Document findings — whether threats are discovered or not — to inform detection improvements.
  • 5. Incident Linkage & Campaign Tracking — When incidents occur, attribute to known threat actors or campaigns where possible. Track campaign evolution over time. Share findings back to intel community.
  • 6. Response Playbook Maintenance — Maintain response playbooks for top threat scenarios. Update based on new TTPs, lessons learned from incidents, and changes to the environment.
  • 7. Threat Landscape Reporting — Produce periodic threat landscape briefings for leadership. Highlight changes in threat actor targeting, new TTPs, and recommended defensive investments.

📥 Inputs

  • Paste sites (Pastebin, Ghostbin) — credential dumps, data leaks
  • Social media platforms — employee oversharing, brand impersonation
  • Domain and SSL certificate transparency logs
  • Public cloud storage (S3 buckets, Azure blobs, GCS) — misconfigured access
  • Job postings — technology stack and security tool disclosures
  • Breach databases and credential leak aggregators
  • WHOIS, DNS records, Shodan/Censys scans

📤 Outputs → Correlation Engine

  • Exposed credential alerts — employee/service credentials found in breaches or pastes
  • Code/secret leak findings — API keys, tokens, config files in public repos
  • External attack surface map — all public-facing assets discoverable by adversaries
  • Brand/impersonation threats — lookalike domains, fake social accounts
  • Technology stack intelligence — what adversaries can learn about the org's defenses
  • Employee exposure reports — PII and professional data available for social engineering

⚙️ Process Steps

  • 1. Scope Definition — Define OSINT collection scope: organization domains, subsidiaries, key personnel, technology keywords, brand terms, IP ranges, and monitored competitors/peers.
  • 2. Automated Collection — Deploy monitoring on: code repos (truffleHog, GitLeaks patterns), paste sites, certificate transparency, domain registrations, social media, cloud storage scanners, and breach databases.
  • 3. External Attack Surface Enumeration — Continuously discover all externally visible assets: subdomains, IPs, open ports, services, technologies. Compare against known asset inventory to identify shadow IT.
  • 4. Validation & False Positive Triage — Validate findings before alerting. Confirm credential leaks are for active accounts. Verify exposed data is legitimate (not honeypots or test data). Classify severity.
  • 5. Threat Contextualization — Correlate OSINT findings with threat intelligence. A leaked credential + an active phishing campaign targeting the org = elevated risk. Link findings to threat actor profiles.
  • 6. Remediation Actions — Credential leaks → force password resets. Secret exposure → rotate keys immediately. Domain impersonation → initiate takedown. Exposed cloud storage → restrict access.
  • 7. Continuous Refinement — Expand monitoring scope based on findings. Add new keywords, domains, and personnel as the organization evolves. Track emerging OSINT sources.

Feeds → Correlation Engine

📥 Inputs

  • Dark web marketplace listings (access-as-a-service, data sales)
  • Underground forum posts and private channels (Telegram, Discord, IRC)
  • Ransomware leak sites and victim blogs
  • Credential combo lists and stealer log marketplaces
  • Initial access broker (IAB) listings
  • Exploit marketplaces — zero-day and N-day sales
  • Hacktivism planning channels
  • Dark web paste/data dump sites

📤 Outputs → Correlation Engine

  • Stolen credential alerts — employee credentials being sold or traded
  • Access sale detections — network/VPN/RDP access to org being listed for sale
  • Data breach evidence — org data appearing on leak sites or marketplaces
  • Ransomware targeting signals — org named on ransomware negotiation or leak sites
  • Threat actor chatter — discussions mentioning the org, its industry, or key personnel
  • Exploit relevance alerts — zero-days for technologies in the org's stack

⚙️ Process Steps

  • 1. Monitoring Scope Configuration — Define keywords, domains, email patterns, executive names, brand terms, and technology identifiers to monitor across dark web sources. Include subsidiary and partner identifiers.
  • 2. Continuous Scanning — Deploy dark web monitoring tools (Flashpoint, DarkOwl, Recorded Future, SpiderFoot) to scan forums, markets, paste sites, leak sites, and messaging channels on a continuous basis.
  • 3. Mention Detection & Triage — When org-related content is detected, auto-classify: (a) direct threat (access sale, data breach), (b) indirect threat (industry targeting, related breach), (c) informational (general chatter). Escalate direct threats immediately.
  • 4. Threat Validation — For direct threats: validate authenticity of claimed data/access. Cross-reference with known breach data, internal logs, and threat intel. Assess if the offering is legitimate or a scam.
  • 5. Impact Assessment — Determine scope: how many credentials, what data types, what access level. Estimate business impact, regulatory notification requirements, and potential adversary capabilities gained.
  • 6. Response Coordination — Credential exposure → mass password reset + MFA enforcement. Access sale → emergency investigation + access revocation. Data breach → legal/compliance notification + IR activation. Threat chatter → elevated monitoring.
  • 7. Intelligence Feedback Loop — Feed dark web findings back into threat management and OSINT workflows. Update threat actor profiles. Refine monitoring keywords based on evolving adversary language and marketplaces.

  • DLP policy configurations and data classification schemas

  • Backup and recovery configurations and test results

  • Control coverage matrix — which controls protect which assets/attack vectors
  • Coverage gaps — assets/segments with missing or degraded controls

  • 2. Coverage Mapping — For each asset class (endpoints, servers, containers, SaaS apps, network segments), document which security controls are deployed. Identify gaps where asset classes lack expected controls.
  • 3. ATT&CK-Based Coverage Assessment — Map every security control to the MITRE ATT&CK techniques it detects or prevents. Generate a heatmap showing defended vs. undefended techniques. Prioritize gaps based on threat model.
  • 4. Effectiveness Validation — Conduct periodic control validation through: purple team exercises, breach and attack simulation (BAS) tools, tabletop exercises, and analysis of controls' performance during real incidents.
  • 5. Health & Deployment Monitoring — Continuously monitor: agent online/offline rates, policy synchronization status, log ingestion volumes (sudden drops = blind spots), and license utilization vs. entitlements.
  • 6. Configuration Drift Detection — Baseline security tool configurations. Alert on policy changes, rule modifications, exclusion additions, and setting changes. Require change approval workflows for security tool configs.
  • 7. Optimization & Rationalization — Identify tool overlap and redundancy. Assess consolidation opportunities. Recommend investments in areas with low coverage or declining effectiveness. Report ROI metrics to leadership.

Security Gap Analysis

Cross-referencing the threat landscape, vulnerability posture, and control coverage to identify where the organization's defenses do not adequately address the risks it faces — the delta between "what we need" and "what we have."

Threat Requirements
→
Current State Assessment
→

→

→
Remediation Roadmap
▼
Feeds → Correlation Engine

📥 Inputs

  • Outputs from all other workflows (threat model, vuln data, control coverage, compliance status)
  • Business risk appetite and risk tolerance statements
  • Prior audit findings and remediation status
  • Industry benchmarking data (peer comparison)
  • Insurance requirements and carrier feedback
  • Board/executive risk directives
  • M&A target security assessment data
  • Regulatory change notifications

📤 Outputs → Correlation Engine

  • Gap register — every identified gap with severity, affected assets, and risk exposure
  • Risk-ranked remediation roadmap — prioritized initiatives with cost, effort, timeline
  • Quick wins list — high-impact, low-effort gaps that can be closed immediately
  • Investment justification packages — business cases for major security investments
  • Residual risk report — risk remaining after planned remediation
  • Maturity progression tracking — gap closure trend over time

⚙️ Process Steps

  • 1. Aggregate Cross-Workflow Data — Pull current state from: threat management (what we face), vulnerability management (what's exposed), controls inventory (what we defend with), compliance (what we must satisfy). This is the foundation — it requires output from every other workflow.
  • 2. Define "Required State" — Based on threat model, risk appetite, regulatory requirements, and industry best practices, define what security capabilities, controls, and maturity levels the org needs. Map to frameworks (NIST CSF, CIS Controls).
  • 3. Current vs. Required Comparison — Systematically compare current state against required state across every domain: identity, endpoint, network, cloud, data, application, incident response, governance. Identify deltas.
  • 4. Gap Severity Assessment — For each gap: what threat actors could exploit it, what assets are at risk, what's the maximum business impact, and what's the likelihood of exploitation given current threat landscape. Use the correlation engine's risk scores.
  • 5. Remediation Option Analysis — For each gap, define 2-3 remediation options ranging from low-cost/compensating to comprehensive/strategic. Estimate cost, effort, timeline, and residual risk for each option.
  • 6. Prioritized Roadmap Construction — Build a phased remediation roadmap: Phase 1 (0-30 days) critical gaps and quick wins, Phase 2 (30-90 days) high-priority, Phase 3 (90-365 days) strategic. Align with budget cycles.
  • 7. Continuous Gap Tracking — Monitor gap closure progress. Re-assess quarterly as threat landscape, business environment, and control posture evolve. Recalibrate roadmap based on new data.

  • 7. Reporting & Trend Analysis — Generate compliance dashboards showing posture across all frameworks. Track maturity progression over time. Report to board on regulatory risk exposure and compliance investment ROI.

⬡ Layer 3 — Unified Risk Scoring & Prioritization

Every correlated finding receives a composite risk score incorporating threat likelihood, vulnerability exposure, control effectiveness, and business impact.

Composite Risk Score Formula

Risk Score = Threat_Likelihood × Vulnerability_Exposure × (1 - Control_Effectiveness) × Business_Impact
Where:
  Threat_Likelihood = f(active threat actors, campaign activity, dark web signals, EPSS)
  Vulnerability_Exposure = f(CVSS, exploit availability, network exposure, asset count)
  Control_Effectiveness = f(control coverage %, tool health, detection validation, defense depth)
  Business_Impact = f(data sensitivity, revenue dependency, regulatory scope, brand exposure)

Backend AI Correlation Pipeline

Purpose: This section defines the technical backend workflow for how the AI correlation engine actually processes data — from raw ingestion through to correlated insights. This is the "how it works under the hood" reference for engineering and AI/ML teams building the platform.

AI Correlation Pipeline — Step-by-Step

1
Data Ingestion & Connector Layer

Standardized connectors pull data from each workflow's output via APIs, webhooks, or scheduled batch jobs. Supported formats: STIX/TAXII for threat intel, SARIF for vuln scanners, CloudEvents for real-time infra alerts, CSV/JSON for vendor assessments. Each connector handles authentication, rate limiting, pagination, and error retry. Raw data lands in a staging data lake (S3/GCS) partitioned by source and timestamp.

API Connectors Webhook Receivers Batch Importers Schema Validators
2
Normalization & Entity Resolution

Raw data is transformed into a unified schema. Entity resolution links references to the same asset, identity, or vulnerability across different sources (e.g., an IP address in a vuln scan, an AWS instance ID in infrastructure data, and a hostname in threat intel all resolve to the same asset entity). Uses fuzzy matching, deterministic rules, and ML-based entity linking. Output: a normalized entity graph stored in a graph database (Neo4j / Neptune).

Schema Mapping Entity Dedup Graph Construction Identity Resolution
3
Enrichment & Context Layering

Each entity in the graph is enriched with business context: asset owner, business unit, data classification, environment (prod/dev), regulatory scope (PCI, HIPAA, GDPR), revenue attribution, and SLA requirements. This context layer transforms raw technical findings into business-relevant risk data. Sources: CMDB, HR directory, data classification catalog, financial systems.

Business Context Asset Ownership Data Classification Regulatory Tagging
4
AI Correlation Engine (Core)

The heart of the system. Runs three correlation mechanisms in parallel:

(a) Rule-Based Correlations: Deterministic rules for known-good correlation patterns (e.g., "IF dark_web_credential_match AND iam_admin_role AND no_mfa THEN critical_alert"). Fast, explainable, low false-positive.

(b) Graph-Based Pattern Detection: Traverses the entity graph to discover multi-hop risk paths (e.g., vendor breach → SBOM vuln → production system → PCI data). Uses graph algorithms (shortest path, centrality, community detection) to find high-risk clusters.

(c) LLM-Powered Contextual Analysis: For nuanced correlations that require understanding context, an LLM analyzes combined findings and generates: natural language risk narratives, non-obvious connection identification, recommended response actions, and executive summary generation. The LLM operates on pre-processed, structured data — not raw feeds.

Rule Engine Graph Traversal LLM Analysis Parallel Execution
5
Risk Scoring & Prioritization

Each correlated finding receives a composite score using the formula from Layer 3 above. The scoring model is a weighted ensemble combining: deterministic scoring (CVSS, EPSS, CWSS), ML-based scoring (trained on historical incident data to predict exploitability), and business impact multipliers (from enrichment layer). Scores are normalized to 0-100 and bucketed into Critical (>85), High (70-85), Medium (40-70), Low (<40). Deduplication ensures the same risk isn't counted multiple times from different correlation paths.
